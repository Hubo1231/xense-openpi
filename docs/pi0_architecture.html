Pi0 模型架构与数据流

本页按源码中的 Pi0.__init__、embed_prefix、embed_suffix、 compute_loss 和 sample_actions 梳理模型。核心思路是：视觉和语言形成 prefix token，状态、噪声动作和时间形成 suffix token，二者进入同一个 Gemma 双专家 transformer：paligemma 处理 prefix，action_expert 处理 suffix， 最后由 action head 预测 flow matching 速度场 v_t。

视觉输入 3 路 RGB base、左腕、右腕，默认 224x224。
语言输入 Prompt tokens Pi0 默认 max_token_len=48，Pi05 默认 200。
动作输出 50 x 32 默认 action_horizon=50，action_dim=32。
Transformer 双专家 Gemma prefix 用 PaliGemma/Gemma，suffix 用 action expert。

新增总体架构图：FastViT 触觉后融合方案

设计思路是保留原来的视觉 / 语言 / 动作主干，把触觉图像单独送入 FastViT 编码，再在 LLM 之后和 suffix_out 做后融合。这样对原始 Pi0 干扰最小，也更容易复用已有权重。 这里画的是你要的新增方案图，不是仓库当前 `pi0_tactile.py` 的既有实现。

视觉走 prefix，触觉走 post-LLM

视觉 触觉 语言 动作/状态/时间 LLM 与融合
1 保留主干
  1. 视觉、语言、状态、动作这条原始 Pi0 链路保持不变。
  2. 这样可以最大限度复用已有的 prefix / suffix / action head 设计。
2 FastViT 触觉分支
  1. 左 / 右触觉图像单独进入 FastViT 编码器。
  2. 输出先池化成 tactile_emb，再用线性层对齐到 LLM 宽度。
3 LLM 后融合
  1. LLM 先只处理原始 prefix / suffix，不让触觉扰动注意力结构。
  2. suffix_out 和 tactile_emb 在后融合层合并，再进 action_out_proj。
推荐的实现形态是 FastViT -> tactile_emb -> concat([suffix_out, tactile_emb]) -> fusion_proj -> action_out_proj。 门控可以作为后续增强，但主方案是先拼接再线性投影。这样触觉是“额外条件”，不是“改写主干”，调试和迁移都更稳。

Pi05 模型总体架构图

Pi05 仍然复用 Pi0 的视觉 / 语言 prefix 与动作 expert，但有两个关键变化：状态不再经过 state_proj 变成连续 suffix token，而是离散化后并入语言 token；flow matching 的时间 t 不再和 action token 拼接，而是通过 time_mlp 生成 adarms_cond， 注入 action expert 的 adaRMSNorm。

Pi05: discrete state + adaRMS time conditioning

视觉 prefix 语言/离散状态 prefix 动作 suffix Gemma 双专家
状态在 prefix
  1. discrete_state_input=True，状态由数据 transform 编进 prompt token。
  2. embed_suffix 不再创建 state_proj(obs.state) 的连续 state token。
动作 suffix 更干净
  1. action_tokens = action_in_proj(x_t) 直接作为 action expert 输入。
  2. suffix 的最后 action_horizon 个 hidden state 全部对应动作 token。
时间走 adaRMS
  1. posemb_sincos(t) -> time_mlp_in/out -> adarms_cond。
  2. PaliGemma.llm(..., adarms_cond=[None, adarms_cond]) 只调制 action expert。

Pi05 中 prefix_tokens / suffix_tokens 的 LLM 前向过程

这张图对应 self.PaliGemma.llm([prefix_tokens, suffix_tokens], ...)。 Pi05 不是把两类 token 简单 concat 后走同一套权重，而是用同一个 Gemma Module 调度两个 expert： paligemma 负责 prefix，action_expert 负责 suffix。attention 在 token 维度联合计算， FFN / norm / 输出投影则按 expert 分开使用各自权重。

Pi05: [prefix_tokens, suffix_tokens] -> dual-expert Gemma -> suffix_out

prefix / paligemma suffix / action_expert joint attention action head
prefix_tokens
  1. embed_prefix 把图像 token 和 prompt/state token 拼成 prefix。
  2. Pi05 的状态已经离散化到 prompt 侧，所以 prefix 包含视觉、语言和离散状态。
suffix_tokens
  1. embed_suffix 中 action_in_proj(x_t) 直接得到 action expert token。
  2. time_mlp(t) 不拼进 token，而是作为 adarms_cond 调制 action expert。
Gemma Block
  1. attention 前两个 expert 分别做 norm 和 Q/K/V 投影，然后在 token 维度 concat 做联合 attention。
  2. attention 输出再切回两个 expert，各自走 output projection、FFN、final norm，返回 prefix_out 和 suffix_out。
对 action head 来说，真正使用的是 suffix_out[:, -action_horizon:]。 它不是原始 action token，而是 action expert 在每层中通过 attention 读取 prefix 上下文、 再经过 action expert 自己的 FFN / adaRMS 后得到的动作隐状态。

Pi05 + FastViT 触觉经 action_expert 融合架构图

这是把 FastViT 输出放进 Pi05 action_expert 的方案：视觉 / 语言 / 离散状态仍走 paligemma prefix；触觉图像经 FastViT 后投影成 tactile suffix tokens， 放在 action tokens 前面，一起进入 action_expert。这样触觉会经过 18 层 action expert， 并在每层 joint attention 中参与动作 token 的计算。

Pi05 主干 + FastViT tactile tokens into action_expert

视觉 语言/离散状态 动作/时间 触觉 LLM / action_expert
Pi05 主干
  1. 状态被前移到 prefix，和 prompt 一起进入 Gemma embedder。
  2. 视觉 / 语言 / 离散状态仍走 paligemma prefix，不把触觉塞进 prefix。
触觉进 suffix
  1. 触觉图像单独走 FastViT，再投影到 action_expert_config.width。
  2. 把 tactile_tokens 放在 action tokens 前面：suffix_tokens=[tactile, action]。
输出切片
  1. suffix_out 同时包含 tactile token hidden states 和 action token hidden states。
  2. action head 只取最后 action_horizon 个 action hidden states。
推荐实现是新增 tactile_proj，把 FastViT 输出对齐到 action expert 宽度，然后在 embed_suffix 中先 append tactile_tokens，再 append action_tokens。 adarms_cond 可以用 token-wise 形式：tactile token 对应零向量或单独的 tactile cond， action token 对应原来的 time_mlp(t)。这样触觉会完整经过 action expert 的 attention / FFN。

Prefix: 图像和语言如何进入模型

视觉 token
  1. preprocess_observation 保证需要的 3 路图像存在，并 resize 到 224x224。
  2. self.PaliGemma.img 使用 SigLIP So400m/14，pool_type="none"。
  3. 每路图像变为 patch token 序列，224/14=16，所以每路约 256 个视觉 token。
  4. num_classes=paligemma_config.width，让图像 token 维度对齐 prefix expert。
语言 token
  1. 如果 obs.tokenized_prompt 存在，调用 PaliGemma.llm(..., method="embed")。
  2. 输出 token embedding 与图像 token 拼接。
  3. tokenized_prompt_mask 直接作为语言 token 的有效位。
prefix mask
  1. 图像 mask 被 repeat 到图像 token 长度。
  2. 图像和语言 token 的 ar_mask 都是 False。
  3. 因此 prefix 内部是双向可见的 full attention。
箭头表达：images -> SigLIP -> image_tokens，prompt ids -> Gemma embedder -> prompt_tokens， [image_tokens, prompt_tokens] -> prefix_tokens。

训练数据流

标准 loss
  1. actions 和 Gaussian noise 按时间 t 插值：x_t = t * noise + (1 - t) * actions。
  2. 目标速度场是 u_t = noise - actions。
  3. embed_prefix(observation) 与 embed_suffix(observation, x_t, t) 同时生成 token。
  4. make_attn_mask 生成 transformer mask，随后一次完整前向得到 suffix_out。
  5. action_out_proj 输出 v_t，loss 是 mean((v_t - u_t)^2)。
RTC 训练 loss
  1. 随机采样 delay，形成 action_prefix_mask。
  2. prefix 动作保持 clean：对应时间设为 0.0。
  3. postfix 动作按随机 t 加噪。
  4. 模型仍预测完整 v_t，但 loss 只统计 postfix 区域。
注意力关系
  1. prefix token 的 ar_mask=False，图像和语言互相可见。
  2. suffix 第一个 token 的 ar_mask=True，切出新 attention block。
  3. suffix 可看 prefix，prefix 不反看 suffix。
  4. action horizon 内部 False 连续，属于同一个 action block。
标准训练箭头：actions + noise + t -> x_t -> suffix_tokens -> Gemma -> suffix_out -> action_out_proj -> v_t -> MSE(v_t, u_t)。

Pi0 与 Pi05 分支差异

位置 Pi0 默认 Pi05
配置开关 pi05=False，模型类型是 ModelType.PI0。 pi05=True，模型类型是 ModelType.PI05。

Pi05 推理时的 RTC 运行图

这张图对应 src/openpi/models/pi0.py 里的 training_time_rtc_sample_actions。 推理时 RTC 分成三件事：客户端把上一段没执行完的动作作为 prev_chunk_left_over 传回； Policy.infer() 把它转换到训练动作空间；模型在每个 denoise step 里重新写回动作 prefix， 只让 postfix 继续按 flow matching 更新。

简化版：Broker -> Policy -> JAX sampler -> Queue

客户端 / 调度 RTC action prefix 观测 prefix KV cache suffix denoise
1 客户端触发
  1. RTCActionChunkBroker 以 runtime 频率消费动作。
  2. 队列剩余动作低于阈值时，向 policy server 发起新 infer 请求。
2 传回 prefix
  1. Policy.infer() 把上一段 leftover 动作重新放进训练时动作空间。
  2. 再把 prev_chunk_left_over 和 inference_delay 传给 training_time_rtc_sample_actions。
3 模型内冻结
  1. 模型每一步 denoise 前都把动作 prefix 写回 x_t，并把 prefix 时间设为 0。
  2. kv_cache 属于观测 prefix cache，和 prev_chunk_left_over 是两条不同路径。
这条 RTC 链路的关键不是“把 old chunk 塞进 LLM prefix”，而是“客户端在管请求节奏，Policy 在管动作空间转换， 模型在管 RTC action prefix 冻结和 postfix denoise”。实际输出的仍然是一个 action chunk， 只是它会随着队列余量、延迟估计和 merge 过程不断滚动更新。
